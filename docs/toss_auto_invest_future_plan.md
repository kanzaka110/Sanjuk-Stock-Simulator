# 토스증권 자동투자 Future Plan

> **상태: 문서(설계 초안)만 존재. 구현 코드 없음.**
> 토스증권 Open API가 발급된 이후를 대비한 future plan이며,
> 지금 당장 어떤 주문/자동매매/API 클라이언트도 구현하지 않는다.

---

## 1. 목적

- 토스증권 Open API가 **발급된 이후**를 대비한 future plan을 미리 정리한다.
- **지금 당장 구현하지 않는다.** 이 문서는 설계 가드레일이며, 실제 토스 API 연동/주문 코드는 포함하지 않는다.
- 삼성증권 **원본 포트폴리오**와 토스 **실험 계좌**를 명확히 분리한다.
- 기존 브리핑 추천을 토스 자동주문에 그대로 직결하지 않는다.

---

## 2. 계좌/자금 분리 원칙

| 구분 | 삼성증권 | 토스증권 |
|------|----------|----------|
| 성격 | **실제 투자 원본 계좌** | 향후 **실험 계좌** |
| 자금 규모 | 현재 보유 전체 | 약 **1,000만원** 규모 |
| 기준 | 기존 보유/브리핑 기준 | 토스 전용 전략/리스크/성과 |
| 운용 | 수동 + 브리핑 의사결정 | (미래) Hermes 자동투자 실험 |

분리 규칙:

- 삼성증권: 실제 투자 원본. 기존 보유/브리핑 추천의 기준 계좌로 **유지**.
- 토스증권: 향후 1,000만원 규모의 **별도 실험 계좌**.
- 성과 / 리스크 / 주문 후보 / 현금 / 포지션을 **별도 DB 또는 별도 namespace**로 분리한다.
- **삼성 브리핑 추천 → 토스 자동주문 직결 금지.** 토스는 독립 전략 필터를 통과한 후보만 다룬다.

---

## 3. 단계별 로드맵

### Phase 0 — 신청 / 승인 대기
- 토스증권 계좌 확인
- Open API 사전 신청
- 승인 / 키 발급 대기
- GCP outbound IP / allowed IP 준비
- 키 저장 정책 문서화 (`.env` / Secret Manager만 사용)
- **공식 승인 SLA는 알 수 없음.** 실무 추정치로만 **2~7영업일** 정도 가정(보장 아님).

### Phase 1 — Read-only probe
- 계좌 조회
- 잔고 조회
- 보유 종목 조회
- 시세 조회
- **주문 endpoint 사용 금지**
- API rate limit / 에러 / 재시도 정책 관찰 및 기록

### Phase 2 — Paper trading
- 토스 계좌 기준 **가상 주문 후보**만 생성
- **실제 주문 없음**
- 기존 브리핑 추천과 분리된 **토스 전용 전략 필터** 적용
- 성과는 **별도 paper ledger**에 기록 (삼성 성과와 혼합 금지)

### Phase 3 — Approval gate
- Telegram 승인 / 거절 버튼 또는 명시 승인 문구 설계
- **기본값은 미주문**
- 주문표 **8필드 완성 필수** (종목/방향/수량/지정가/목표가/손절/취소조건/계좌)
- 일 / 종목 / 계좌 / 금액 제한 적용

### Phase 4 — Limited live order 후보
- **아직 구현 금지**라고 명시한다.
- 충분한 paper 성과와 수동 승인 검증 이후에만 **별도 작업으로** 진행한다.
- 주문 전 **Hermes 독립 검증 필수**.

---

## 4. 리스크 게이트

- 1일 최대 주문금액 상한
- 1종목 최대 비중 상한
- 총 현금 비중 하한
- 손실 **연속 발생** 시 자동 중단(킬 스위치)
- API 오류 / 시세 stale / 환율 stale 시 **주문 차단**
- MU 같은 **보호 종목**은 토스 전략에도 **별도 보호 / 보유관리 정책** 필요 (실행 매도 아님)
- **지정가 기준, 시장가 금지** 또는 극히 제한

---

## 5. 기술 설계 원칙

- `core/toss_*` 같은 future module 후보명은 **문서에만 적고 생성하지 않는다.**
  - 예시 후보명(미생성): `core/toss_client`, `core/toss_ledger`, `core/toss_strategy`, `core/toss_order` 등
- **read-only GET / probe부터** 시작한다.
- order client는 **별도 인터페이스로 격리**한다 (시세/조회 클라이언트와 분리).
- **dry-run 기본값.** 실주문은 명시적으로 켜야만 동작.
- 실주문 기능은 **feature flag + 승인 게이트 + 별도 테스트 + 별도 커밋**에서만 도입한다.
- secrets는 `.env` / Secret Manager에만 둔다. **문서 / 로그 / 테스트에 절대 노출 금지.**
- API 키 노출 시 **즉시 폐기 / 재발급**.

---

## 6. 테스트 / 검증 계획

- 주문 endpoint가 없는지 grep으로 확인
- POST / PUT / DELETE가 없는지 grep으로 확인
- dry-run 결과와 주문표 스냅샷 테스트
- paper ledger 테스트
- rate limit / 오류 응답 테스트
- 실주문 전 체크리스트 통과 확인

---

## 7. 명시적 Non-goals

- 이번 PR / 커밋에서 **토스 API 클라이언트 구현 안 함**
- **주문 endpoint 구현 금지** — 주문 실행 안 함
- **실제 주문 구현 금지** — 자동매매 안 함
- 삼성증권 포트폴리오 원본 대체 안 함
- 기존 브리핑 추천 자동 주문 연결 안 함
